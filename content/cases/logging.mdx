## Problem

Travel-Maker 운영 중 에러가 발생했을 때 원인을 파악하기 어려운 상황이 반복됐습니다.

Django와 Celery 로그가 콘솔에만 출력되고 있었기 때문에, 서버를 재시작하거나 시간이 조금만 지나면 이전 로그가 사라졌습니다. 1주일 전 에러는 물론 하루 전 에러도 추적이 불가능했습니다.

## Investigation

로깅 설정을 확인하니 별도 파일 핸들러 없이 콘솔 출력(`StreamHandler`)만 구성되어 있었습니다. 에러가 발생했을 때의 흐름을 재현하려면 같은 상황을 다시 만들어야 했고, 이는 사실상 불가능한 경우가 많았습니다.

Celery worker 역시 마찬가지였습니다. 태스크가 실패해도 로그가 남지 않아 어떤 태스크에서, 어떤 이유로 실패했는지 알 수 없었습니다.

## 고민한 방법들

**① Sentry 같은 외부 에러 트래킹 서비스 도입**
에러 발생 시 알림과 스택 트레이스를 제공하는 강력한 도구입니다. 하지만 무료 플랜의 이벤트 제한이 있고, 에러 외의 일반 로그(요청/응답 흐름 등)는 추적할 수 없습니다. 지금 당장 필요한 건 에러 트래킹보다 "흐름을 기록하는 체계"였습니다.

**② 콘솔 출력을 systemd/nohup으로 파일에 리다이렉트**
추가 코드 없이 파일로 남길 수 있지만, 로테이션이 없어 시간이 지나면 디스크가 가득 차는 문제가 생깁니다. 로그 파일 관리를 별도로 해야 하는 번거로움도 있습니다.

**③ TimedRotatingFileHandler (채택)**
Python 표준 라이브러리에 포함된 핸들러로 추가 의존성이 없습니다. 날짜별로 파일을 분리 저장하고 지정한 보존 기간(7일)이 지난 파일은 자동으로 삭제합니다. 디스크 관리와 로그 추적을 동시에 해결할 수 있어 선택했습니다.

## Solution

`TimedRotatingFileHandler`를 Django와 Celery 로그에 각각 적용했습니다.

```python
# settings.py
LOGGING = {
    'version': 1,
    'handlers': {
        'file': {
            'class': 'logging.handlers.TimedRotatingFileHandler',
            'filename': BASE_DIR / 'logs/django.log',
            'when': 'midnight',
            'backupCount': 7,
            'formatter': 'verbose',
        },
    },
    'root': {
        'handlers': ['file'],
        'level': 'INFO',
    },
}
```

Celery도 동일한 핸들러를 연결해 워커 태스크 실행 흐름과 실패 로그를 별도 파일로 분리 저장했습니다. Celery Flower도 함께 도입해 워커 상태를 실시간으로 모니터링할 수 있는 대시보드를 구성했습니다.

## Result

에러 발생 시점과 흐름을 실시간/사후로 모두 추적할 수 있게 됐습니다. 7일치 로그가 날짜별 파일로 보존되면서 디스크 사용량도 일정 수준 이하로 유지됩니다.
