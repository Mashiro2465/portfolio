## Problem

OZ Externship(LMS) 프로젝트의 질문 목록 조회 API에서 각 질문마다 대/중/소 카테고리 경로를 함께 표시해야 했습니다.

구현 당시 `category.parent.parent` 형태로 중첩 관계 필드에 접근했는데, 목록 조회 시 질문 수만큼 쿼리가 반복 발생하는 N+1 문제가 있었습니다.

## Investigation

Django ORM은 기본적으로 **지연 로딩(Lazy Loading)** 방식으로 동작합니다. ForeignKey 관계 필드에 접근할 때마다 별도의 `SELECT` 쿼리가 발생합니다.

질문 목록이 N개라면:
- 질문 목록 조회: 1개 쿼리
- 각 질문의 소분류 조회: N개 쿼리
- 각 소분류의 중분류 조회: N개 쿼리
- 각 중분류의 대분류 조회: N개 쿼리
- 각 질문의 작성자 조회: N개 쿼리

결과적으로 **4N + 1개**의 쿼리가 발생하고 있었습니다.

```python
# 문제가 된 코드 (쿼리 폭발)
for question in Question.objects.all():
    category_path = f"{question.category.parent.parent.name} > {question.category.parent.name} > {question.category.name}"
```

## 고민한 방법들

**① `prefetch_related` 사용**
`prefetch_related`는 관련 객체를 별도 쿼리로 한 번에 가져와 Python 레벨에서 매핑합니다. M2M이나 역방향 관계(ForeignKey의 related_manager)에 적합한 방식입니다. 그러나 `category → parent → parent` 처럼 ForeignKey 순방향으로 깊이 타고 들어가는 경우에는 각 depth마다 쿼리가 분리되어 오히려 쿼리 수가 늘 수 있습니다.

**② `select_related` 사용 (채택)**
ForeignKey 순방향 관계를 SQL JOIN으로 한 번에 가져옵니다. `select_related("category__parent__parent", "author")`처럼 경로를 명시하면 depth가 깊어도 JOIN 한 번으로 처리됩니다. 카테고리 계층 탐색처럼 단일 객체를 따라 올라가는 패턴에는 `select_related`가 정확히 맞는 도구입니다.

## Solution

`select_related`로 관련 데이터를 JOIN 쿼리 하나로 한 번에 로딩하고, 답변 수 같은 집계 값은 `annotate`로 처리했습니다.

```python
from django.db.models import Count, Subquery, OuterRef

queryset = Question.objects.select_related(
    "category__parent__parent",  # 소분류 → 중분류 → 대분류 JOIN
    "author"
).annotate(
    answer_count=Count('answers'),
    thumbnail=Subquery(
        QuestionImage.objects.filter(
            question=OuterRef('pk')
        ).values('image_url')[:1]
    )
)
```

`select_related`는 INNER/LEFT OUTER JOIN으로 관련 테이블을 한 번에 가져오므로 중첩 depth가 깊어도 쿼리 수는 고정됩니다.

## Result

4N + 1개였던 반복 쿼리가 일정한 수의 쿼리로 고정되었습니다. 목록 크기가 커질수록 효과가 커지며, 응답 시간도 눈에 띄게 개선되었습니다.
