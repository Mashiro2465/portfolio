## Problem

Travel-Maker의 이미지 업로드 기능은 클라이언트가 AWS S3에 직접 `PUT` 요청을 보내는 구조였습니다.

이 방식은 클라이언트 코드에 AWS Access Key ID와 Secret Access Key를 포함해야 합니다. 프론트엔드 코드는 브라우저에 노출되므로, 자격증명이 외부에 유출되면 S3 버킷 전체에 대한 무단 접근이 가능해지는 심각한 보안 위험이 있었습니다.

## Investigation

기존 업로드 플로우를 분석했습니다.

- 클라이언트가 AWS SDK를 직접 사용해 S3에 PUT 요청
- AWS 자격증명이 환경 변수 또는 빌드 설정에 포함되어 프론트엔드 번들에 포함될 가능성

이를 Pre-signed URL 방식으로 전환하는 과정에서 추가 문제가 발생했습니다. S3 유틸리티 모듈을 분리하는 과정에서 Django 앱 간 `import` 경로 충돌이 일어났고, 이를 모듈 재구성으로 함께 해결해야 했습니다.

## 고민한 방법들

**① 백엔드 경유 업로드 (클라이언트 → 백엔드 → S3)**
클라이언트가 백엔드에 파일을 올리면 백엔드가 S3에 대신 저장하는 방식입니다. 자격증명이 서버에만 있어 보안 문제는 해결됩니다. 하지만 파일이 서버 메모리를 거쳐야 하므로, t2.micro(1GB RAM)에서 이미지가 많이 올라오는 상황이면 메모리 압박이 심해집니다. 이미 OOM 문제를 겪고 있는 환경에서 업로드 트래픽까지 서버가 담당하는 건 무리라고 판단했습니다.

**② Pre-signed URL 방식 (채택)**
백엔드는 임시 서명된 URL만 발급하고, 실제 파일 전송은 클라이언트가 S3에 직접 합니다. 자격증명은 서버 밖으로 나가지 않고, 업로드 트래픽도 서버를 우회합니다. URL 유효 기간을 1시간으로 제한해 탈취되더라도 피해 범위가 좁습니다. 보안과 서버 부담 모두 해결할 수 있어 선택했습니다.

## Solution

**Pre-signed URL 방식으로 전환**했습니다.

백엔드에서 boto3를 통해 임시 서명된 업로드 URL을 발급하고, 클라이언트는 해당 URL로만 S3에 직접 PUT 요청을 보내도록 변경했습니다. AWS 자격증명은 서버에만 존재하며 클라이언트에는 절대 노출되지 않습니다.

```python
import boto3
from django.conf import settings

def generate_presigned_upload_url(key: str, expiration: int = 3600) -> str:
    s3 = boto3.client(
        's3',
        aws_access_key_id=settings.AWS_ACCESS_KEY_ID,
        aws_secret_access_key=settings.AWS_SECRET_ACCESS_KEY,
        region_name=settings.AWS_S3_REGION_NAME,
    )
    return s3.generate_presigned_url(
        'put_object',
        Params={'Bucket': settings.AWS_STORAGE_BUCKET_NAME, 'Key': key},
        ExpiresIn=expiration,
    )
```

업로드 플로우:
1. 클라이언트가 백엔드에 Pre-signed URL 발급 요청
2. 백엔드가 임시 URL 생성 후 반환 (유효 기간 1시간)
3. 클라이언트가 해당 URL로 직접 S3에 파일 업로드

모듈 import 충돌은 S3 관련 유틸리티를 별도 앱에서 공용 모듈로 분리해 해결했습니다.

## Result

AWS 자격증명이 클라이언트에 전혀 노출되지 않는 안전한 업로드 구조를 확보했습니다. URL 유효 기간이 1시간으로 제한되어 탈취되더라도 피해 범위가 최소화됩니다.
