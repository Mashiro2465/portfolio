## Problem

Travel-Maker 프로젝트 초기에는 EC2 서버에서 직접 `docker build`를 실행하는 방식으로 배포하고 있었습니다.

t2.micro 인스턴스의 RAM은 1GB로, Django 애플리케이션을 실행하는 것만으로도 메모리가 빠듯한 환경이었습니다. 여기에 Docker 이미지 빌드까지 병행하면 빌드 중 메모리가 급증하여 OOM Killer가 프로세스를 강제 종료하거나, GitHub Actions에서 SSH로 빌드 명령을 실행할 때 타임아웃이 반복적으로 발생했습니다.

## Investigation

배포 실패 로그를 분석한 결과 두 가지 패턴을 확인했습니다.

1. **OOM**: `docker build` 중 메모리 사용량이 임계치를 초과해 OOM Killer가 빌드 프로세스를 종료
2. **타임아웃**: SSH 세션을 통해 장시간 빌드 명령을 실행하면 GitHub Actions의 커넥션이 끊기거나 작업 시간 제한에 걸림

근본 원인은 빌드 부하를 실행 서버가 함께 담당하는 구조 자체에 있었습니다. t2.micro는 애플리케이션 실행 전용으로만 사용해야 하는 환경이었습니다.

## 고민한 방법들

세 가지 선택지를 검토했습니다.

**① EC2 인스턴스 업그레이드 (t2.small 이상)**
가장 빠른 해결책처럼 보였지만, 빌드와 실행을 같은 서버에서 계속 담당하는 구조 자체는 그대로입니다. 트래픽이 늘거나 의존성이 무거워질수록 같은 문제가 반복될 것이고, 비용도 계속 올라갑니다. 근본 해결이 아니라고 판단했습니다.

**② Swap 증설 후 현행 방식 유지**
Swap을 늘리면 OOM은 줄일 수 있지만, GitHub Actions에서 SSH로 장시간 빌드 명령을 실행하는 구조상 타임아웃 문제는 남습니다. 이 역시 임시방편이라 제외했습니다.

**③ GitHub Actions에서 빌드 → Docker Hub push → EC2 pull (채택)**
빌드 부하를 실행 환경에서 완전히 분리하는 방식입니다. EC2는 이미지를 pull해서 실행하기만 하므로 메모리 부담이 없고, SSH 연결 시간도 짧아 타임아웃 위험도 사라집니다. Docker Hub 무료 플랜으로 추가 비용 없이 구성할 수 있다는 점도 고려했습니다.

## Solution

**GitHub Actions → Docker Hub push → EC2 pull** 구조로 전환했습니다.

이미지 빌드는 GitHub Actions의 러너(별도 환경)에서 수행하고, 완성된 이미지를 Docker Hub에 push합니다. EC2는 이미지를 pull해서 컨테이너를 실행하는 역할만 담당하게 바꿨습니다.

```yaml
# .github/workflows/deploy.yml
- name: Build and push image
  run: |
    docker build -t ${{ secrets.DOCKERHUB_USERNAME }}/travel-maker:latest .
    docker push ${{ secrets.DOCKERHUB_USERNAME }}/travel-maker:latest

- name: Deploy to EC2
  uses: appleboy/ssh-action@master
  with:
    script: |
      docker pull ${{ secrets.DOCKERHUB_USERNAME }}/travel-maker:latest
      docker-compose up -d --no-build
```

## Result

반복되던 OOM과 배포 타임아웃이 완전히 해소되었습니다. EC2 서버는 실행만 담당하게 되어 배포 중 서비스 중단 없이 안정적으로 운영할 수 있게 되었습니다.
