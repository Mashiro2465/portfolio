## Problem

OZ Externship(LMS) 프로젝트에서 카테고리는 대분류 → 중분류 → 소분류의 3단계 구조로 설계되었습니다.

모델은 `parent_id`를 사용하는 자기참조(Self-referential) 구조였는데, DB 레벨에는 깊이에 대한 제약이 없었습니다. 따라서 대-소분류만 연결된 2단계 카테고리나 4단계 이상의 카테고리도 물리적으로 생성 가능한 상태였습니다. 데이터 정합성이 API 레벨에서도 보장되지 않고 있었습니다.

## Investigation

카테고리 모델을 살펴봤습니다.

```python
class Category(models.Model):
    name = models.CharField(max_length=100)
    category_type = models.CharField(choices=[('LARGE', ...), ('MIDDLE', ...), ('SMALL', ...)])
    parent = models.ForeignKey('self', null=True, blank=True, on_delete=models.CASCADE)
```

`category_type`과 `parent`는 각자 존재하지만 둘 사이의 관계를 검증하는 로직이 없었습니다. 예를 들어 `category_type=SMALL`이지만 `parent`가 대분류를 가리키는 잘못된 데이터가 생성될 수 있었습니다.

## 고민한 방법들

**① DB 레벨 CHECK 제약 조건 추가**
가장 근본적인 방법이지만, 자기참조 관계에서 부모의 `category_type`을 참조하는 CHECK 제약은 일반적인 SQL로 표현하기 어렵습니다. 트리거를 써야 하는데, Django ORM과의 궁합이 나쁘고 DBMS에 로직이 분산돼 유지보수가 어려워집니다.

**② `Model.clean()` 또는 `save()` 재정의**
모델 레벨에서 검증하면 Admin과 API 양쪽에서 동작합니다. 그러나 DRF에서 `serializer.save()`를 호출할 때 `full_clean()`이 자동으로 호출되지 않아 별도로 호출 코드를 추가해야 합니다. 누락될 경우 검증이 우회될 위험이 있습니다.

**③ Serializer `validate()` (채택)**
DRF에서 `is_valid()`를 호출하면 `validate()`가 반드시 실행되므로 API 레이어에서 검증이 항상 보장됩니다. 커스텀 예외로 명확한 에러 메시지도 반환할 수 있고, 기존 코드 구조를 크게 바꾸지 않아도 됩니다. 이미 Serializer 단에서 입력 검증을 담당하고 있으니 책임 위치도 자연스럽다고 판단했습니다.

## Solution

`AdminCategoryCreateSerializer.validate()`에서 `category_type`과 부모의 실제 `category_type`을 함께 검증하는 로직을 추가했습니다. 규칙 위반 시 커스텀 예외를 발생시켜 명확한 에러 메시지를 반환했습니다.

```python
def validate(self, data):
    category_type = data.get('category_type')
    parent = data.get('parent')

    if category_type == 'LARGE' and parent is not None:
        raise ValidationError("대분류는 부모 카테고리를 가질 수 없습니다.")

    if category_type == 'MIDDLE':
        if parent is None or parent.category_type != 'LARGE':
            raise InvalidMiddleParentException()

    if category_type == 'SMALL':
        if parent is None or parent.category_type != 'MIDDLE':
            raise InvalidSmallParentException()

    return data
```

규칙:
- 대분류: 부모 없음
- 중분류: 부모는 반드시 대분류
- 소분류: 부모는 반드시 중분류

## Result

생성 단계에서부터 계층 구조가 강제되어 임의 깊이의 잘못된 카테고리 생성이 원천 차단되었습니다. 명확한 커스텀 예외로 인해 잘못된 요청에 대한 에러 메시지도 구체적으로 반환할 수 있게 됐습니다.
